# GDELT Kafka Ingestion Service

This branch adds a Docker-based GDELT ingestion service for the NL2Pipeline project. It downloads real GDELT event files, converts each TSV row into a clean JSON message, and publishes the messages into a Kafka topic called `gdelt-events`.

## What this module does

```text
GDELT TSV files
      ↓
Python Producer
      ↓
Kafka topic: gdelt-events
      ↓
Consumer / Spark pipeline / Test harness
```

This gives the project a real streaming data source that can later be used by Spark jobs or LLM-generated big-data pipelines.

## File explanation

| File                      | Purpose                                                                                                          |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `docker-compose.yml`      | Defines Kafka, topic creator, producer, downloader, and consumer test services.                                  |
| `Dockerfile`              | Builds the Python container used by the downloader, producer, topic creator, and consumer.                       |
| `requirements.txt`        | Lists Python dependencies such as `kafka-python`, `requests`, and `PyYAML`.                                      |
| `.env.example`            | Example environment configuration. Copy this to `.env` before running.                                           |
| `.gitignore`              | Prevents local environment files, cache files, and downloaded data from being committed.                         |
| `config/environment.yaml` | Defines the Kafka message schema and maps GDELT column indexes to JSON fields.                                   |
| `app/settings.py`         | Reads environment variables such as Kafka address, topic name, producer mode, and publish rate.                  |
| `app/logging_config.py`   | Sets up consistent logging format for all services.                                                              |
| `app/gdelt_mapper.py`     | Converts raw GDELT TSV rows into JSON messages with fields like `event_id`, `country`, `lat`, `lon`, and `tone`. |
| `app/downloader.py`       | Downloads historic GDELT files in 15-minute intervals and extracts them into the Docker volume.                  |
| `app/create_topic.py`     | Creates the Kafka topic `gdelt-events` if it does not already exist.                                             |
| `app/producer.py`         | Reads GDELT files and publishes JSON messages to Kafka in replay or live mode.                                   |
| `app/consumer_test.py`    | Test script that reads messages from Kafka and prints them to verify that ingestion is working.                  |
| `app/__init__.py`         | Makes the `app` folder a Python package.                                                                         |

## Message schema

Each Kafka message published to `gdelt-events` follows this JSON format:

```json
{
  "event_id": "string",
  "event_date": "YYYY-MM-DD",
  "actor1": "string or null",
  "actor2": "string or null",
  "event_code": "string",
  "country": "string or null",
  "lat": 0.0,
  "lon": 0.0,
  "tone": 0.0,
  "num_mentions": 0,
  "num_sources": 0,
  "num_articles": 0,
  "is_root_event": true,
  "ts": "producer ingest timestamp"
}
```

The `ts` field is generated by the producer when the event is ingested.

## Environment variables

Main values in `.env`:

```env
KAFKA_BOOTSTRAP=kafka:9092
KAFKA_TOPIC=gdelt-events

PRODUCER_MODE=replay
PUBLISH_RATE=10
DATA_PATH=/data/gdelt
REPLAY_LOOP=true

LIVE_POLL_SECONDS=900

START_DATE=20260615000000
END_DATE=20260615010000

LOG_LEVEL=INFO
CONSUMER_MAX_MESSAGES=10
```

| Variable                | Meaning                                                                 |
| ----------------------- | ----------------------------------------------------------------------- |
| `KAFKA_BOOTSTRAP`       | Kafka address used inside Docker. Keep this as `kafka:9092`.            |
| `KAFKA_TOPIC`           | Kafka topic where GDELT events are published.                           |
| `PRODUCER_MODE`         | Use `replay` for historic files or `live` for latest GDELT updates.     |
| `PUBLISH_RATE`          | Number of events published per second.                                  |
| `DATA_PATH`             | Path inside Docker where GDELT files are stored.                        |
| `REPLAY_LOOP`           | If `true`, replay mode loops through files continuously.                |
| `LIVE_POLL_SECONDS`     | How often live mode checks for new GDELT files. Default is 900 seconds. |
| `START_DATE`            | Start timestamp for historic GDELT download. Format: `YYYYMMDDHHMMSS`.  |
| `END_DATE`              | End timestamp for historic GDELT download. Format: `YYYYMMDDHHMMSS`.    |
| `CONSUMER_MAX_MESSAGES` | Number of Kafka messages printed by the test consumer.                  |

## Date format

GDELT files are published every 15 minutes.

The date format is:

```text
YYYYMMDDHHMMSS
```

Example:

```env
START_DATE=20260615000000
END_DATE=20260615010000
```

This downloads files for:

```text
15 June 2026 00:00
15 June 2026 00:15
15 June 2026 00:30
15 June 2026 00:45
15 June 2026 01:00
```

For testing, use a small date range first.

## How to run

### 1. Download GDELT data

Run this once before replay mode:

```bash
docker compose --profile init run --rm gdelt-downloader
```

This downloads and extracts GDELT TSV files into a named Docker volume called `gdelt-data`.

The volume persists between runs. It is only removed if you run:

```bash
docker compose down -v
```

---

### 2. Start Kafka and the producer

```bash
docker compose up --build
```

Expected logs:

```text
gdelt-create-topic | Topic created: gdelt-events
gdelt-producer     | Producer mode: replay
gdelt-producer     | Connected to Kafka at kafka:9092
gdelt-producer     | Starting replay mode
gdelt-producer     | Reading replay file: ...
```

The producer will publish messages into the Kafka topic.

---

### 3. Test Kafka messages

Open a second terminal and run:

```bash
docker compose --profile tools run --rm consumer-test
```

Expected output:

```json
{
  "event_id": "...",
  "event_date": "2026-06-15",
  "actor1": "...",
  "actor2": "...",
  "event_code": "...",
  "country": "...",
  "lat": 0.0,
  "lon": 0.0,
  "tone": 0.0,
  "num_mentions": 0,
  "num_sources": 0,
  "num_articles": 0,
  "is_root_event": true,
  "ts": "2026-06-15T..."
}
```

If JSON messages are printed, the ingestion service is working.

## Replay mode

Replay mode reads previously downloaded GDELT files from the Docker volume and publishes them into Kafka.

Use this in `.env`:

```env
PRODUCER_MODE=replay
PUBLISH_RATE=10
```

Then run:

```bash
docker compose up
```

Replay mode is used for development and testing because the same historic data can be replayed repeatedly.

---

## Live mode

Live mode checks GDELT's `lastupdate.txt` file and streams the latest available export file.

Use this in `.env`:

```env
PRODUCER_MODE=live
LIVE_POLL_SECONDS=900
```

Then restart:

```bash
docker compose up
```

Live mode is mainly for demos. The producer checks for new GDELT files every 15 minutes.

## Common issues

### Docker cannot find Kafka image

If Docker fails to pull a Kafka image, check the image name in `docker-compose.yml`.

This setup uses:

```yaml
image: apache/kafka:3.7.0
```

### Producer says no GDELT files found

Run the downloader first:

```bash
docker compose --profile init run --rm gdelt-downloader
```

### Consumer prints no messages

Make sure the producer is running and publishing events:

```bash
docker compose logs producer
```

Also check that the topic exists:

```bash
docker exec -it gdelt-kafka /opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 --list
```

### Do not commit `.env`

Only commit `.env.example`.

Check before committing:

```bash
git status
```

## Why this helps the NL2Pipeline project

This module provides a real Kafka stream for the rest of the NL2Pipeline system.

Later, generated Spark pipelines can consume from:

```text
Kafka bootstrap: kafka:9092
Topic: gdelt-events
Format: JSON
```

This allows the project to test whether LLM-generated big-data pipeline code can actually connect to Kafka, parse real data, process it, and produce correct output.
